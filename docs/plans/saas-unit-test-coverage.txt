# SaaS Connector Framework: Unit Test Coverage Plan

## Problem Statement

Fidesplus has **301 integration tests across 92 SaaS connectors**, all marked `@integration_saas`. These tests make real HTTP calls to vendor APIs, require active vendor accounts, and depend on 1Password credentials. The result:

- **78 tests permanently skipped** (no active accounts/credentials)
- **29 tests marked xfail** (flaky due to rate limiting or API instability)
- Only ~194 tests are realistically runnable
- Tests are slow, brittle, and expensive to maintain

**Root cause:** These integration tests are overwhelmingly testing the *framework*, not vendor-specific behavior. All 301 tests exercise the same 5 behaviors of the Fides SaaS connector engine through different config shapes:

| Framework Behavior | Integration Tests | Fides Entry Point |
|---|---|---|
| Connection testing | 51 | `SaaSConnector.test_connection()` |
| Data access/retrieval | 42+ | `SaaSConnector.retrieve_data()` |
| Data erasure/masking | 37+ | `SaaSConnector.mask_data()` |
| Consent propagation | 6 | `SaaSConnector.run_consent_request()` |
| Custom request overrides | 2 | `SaaSRequestOverrideFactory` |

The vendor-specific part is the config YAML; the execution engine is identical. If we properly unit-test the engine with representative config combinations, we can catch regressions in seconds instead of relying on 92 flaky vendor accounts.

| Component | Test File | Coverage |
|---|---|---|

| Request override registration/validation | `test_saas_request_override_factory.py` | Good |
| Consent request filtering/identity matching | `test_saas_connector.py` (25+ tests) | Good |

### What's NOT Covered by Unit Tests

| Gap | Current Coverage | Why It Matters |
|---|---|---|
| Full `retrieve_data` pipeline | 40% - individual pieces mocked, never the integrated chain | This is the core read path every SaaS integration uses |
| Full `mask_data` pipeline | 40% - only async/edge cases, no basic UPDATE/DELETE flow | This is the core erasure path |
| `RateLimiter` class | 0% unit tests (only integration tests with real Redis) | Rate limiting bugs silently break all connectors |
| `GET_CONSENTABLE_ITEMS` flow | 0% | Notice-based consent retrieving consentable items |
| `PROCESS_CONSENT_WEBHOOK` flow | 0% | Webhook-driven consent processing |
| `execute_prepared_request` pagination loop | No isolated tests | The pagination-while-loop is the most complex control flow in `retrieve_data` |
| Error handling across full pipelines | Scattered | HTTP errors, postprocessing failures, missing params in the integrated flow |

### Codegraph Confirmation

Codegraph reports **0 direct or transitive test coverage** for the `SaaSConnector` class itself (node: `fides:src/fides/api/service/connectors/saas_connector.py:SaaSConnector`). This confirms the tests in `test_saas_connector.py` test individual static/helper methods but don't exercise the pipeline entry points (`retrieve_data`, `mask_data`, `run_consent_request`) as integrated flows.

`RateLimiter` has only 3 tests, all in `tests/ops/integration_tests/limiter/test_rate_limiter.py` (integration, not unit).

## Test Infrastructure

The existing `MockAuthenticatedClient` in `tests/ops/test_helpers/saas_test_utils.py` is the foundation for all new tests. It:

- Maps `(method, path, query_params)` tuples to `MockResponse` objects
- Supports sequential response stacking (for pagination testing)
- Handles both JSON and binary responses
- Simulates error responses with proper `ClientUnsuccessfulException` raising
- Respects `ignore_errors` behavior

No new test infrastructure is needed. All proposed tests use `MockAuthenticatedClient` + standard pytest mocking.

## Implementation Plan

### Story 1: `retrieve_data` Pipeline Unit Tests

**Repo:** Fides
**Type:** Story
**Estimate:** M
**File:** `tests/ops/service/connectors/test_saas_connector_retrieve_data.py` (new)

Test the full `retrieve_data` flow: `query_config.get_read_requests_by_identity()` -> `query_config.generate_requests()` -> `client.send()` -> `_handle_errored_response` -> `_unwrap_response_data` -> `process_response_data` (postprocessors) -> pagination loop -> `_apply_output_template` -> return rows.

Each test constructs a `SaaSConnector` with an in-memory `SaaSConfig`, wires up `MockAuthenticatedClient`, and calls `retrieve_data()` directly.

**Tests (~12):**

| Test | What It Validates |
|---|---|
| `test_single_collection_access_by_email` | Basic happy path: one collection, one identity, one HTTP call, rows returned |
| `test_single_collection_access_by_phone` | Same flow with phone identity (different `param_value.identity` resolution) |
| `test_multi_read_request_union` | Endpoint with 2 read requests (e.g. active + closed tickets) returning merged rows |
| `test_data_path_extraction` | Response wrapped in `{"data": {"items": [...]}}`, `data_path: "data.items"` unwraps correctly |
| `test_postprocessor_chain` | Unwrap postprocessor followed by filter postprocessor, verifying chaining |
| `test_offset_pagination_full_pipeline` | Offset pagination: mock returns 2 pages, verifies both pages collected via the while-loop |
| `test_cursor_pagination_full_pipeline` | Cursor pagination: mock returns page with cursor, then final page without cursor |
| `test_link_pagination_full_pipeline` | Link pagination: mock returns `Link` header with next URL |
| `test_ignore_errors_true_returns_empty` | HTTP 404 with `ignore_errors: true` returns empty rows, not an exception |
| `test_ignore_errors_specific_codes` | `ignore_errors: [404, 403]` ignores those codes but raises on 500 |
| `test_skip_missing_param_values` | Read request with unresolvable placeholder skips gracefully when flag is set |
| `test_delete_only_endpoint_returns_empty_row` | Endpoint with delete request but no read request returns `[{}]` |

**Acceptance Criteria:**
- All tests use `@pytest.mark.unit_saas` marker
- No database, Redis, or external HTTP calls required
- Tests run in < 2 seconds total

### Story 2: `mask_data` Pipeline Unit Tests

**Repo:** Fides
**Type:** Story
**Estimate:** M
**File:** `tests/ops/service/connectors/test_saas_connector_mask_data.py` (new)

Test the full `mask_data` flow: `query_config.get_masking_request()` -> unwrap rows via `data_path` -> `process_response_data` (postprocessors) -> for each row: `query_config.generate_update_stmt()` -> `client.send()` -> return `rows_updated` count.

**Tests (~8):**

| Test | What It Validates |
|---|---|
| `test_update_masking_basic` | Basic UPDATE masking: 1 row, field replaced with masked value, HTTP PUT sent, returns 1 |
| `test_delete_masking` | DELETE masking: endpoint uses HTTP DELETE, row removed, returns 1 |
| `test_mask_multiple_rows` | 3 rows from access, each triggers separate update request, returns 3 |
| `test_data_path_unwrap_before_masking` | `masking_request.data_path` unwraps nested rows before processing |

| `test_masking_postprocessor_failure_nonfatal` | Postprocessor on the masking response throws, but `rows_updated` still reflects success |

**Acceptance Criteria:**
- All tests use `@pytest.mark.unit_saas` marker
- Verify both the count and the HTTP requests sent to `MockAuthenticatedClient`

### Story 6: `execute_prepared_request` Pagination Loop Tests

**Repo:** Fides
**Type:** Story
**Estimate:** S
**File:** Include in Story 1's test file or as a separate class

The `execute_prepared_request` method is the inner loop of `retrieve_data`. It calls `client.send()`, handles errors, unwraps the response, runs postprocessors, and invokes the pagination strategy. Testing it in isolation (separate from the full `retrieve_data` pipeline) allows testing edge cases in the pagination control flow.

**Tests (~4):**

| Test | What It Validates |
|---|---|
| `test_pagination_terminates_on_empty_page` | Pagination strategy returns None when response has no more data |
| `test_pagination_handles_204_no_content` | 204 response doesn't crash postprocessors or pagination |
| `test_postprocessor_exception_propagates` | PostProcessingException raised during read is not swallowed (unlike masking) |
| `test_response_data_types` | Dict response becomes `[dict]`, list response stays `[dict, dict, ...]`, non-dict raises |

## Testing Strategy

All new tests:
- Use `@pytest.mark.unit_saas` marker
- Use `MockAuthenticatedClient` from `tests/ops/test_helpers/saas_test_utils.py`
- Run without Docker, Redis, Postgres, or external services
- Complete in < 5 seconds total
- Follow existing test patterns in `test_saas_connector.py` for fixture setup

## What This Does NOT Replace

Some fidesplus integration tests remain valuable even after this work:

- **Smoke tests for 2-3 high-value connectors** (e.g. Stripe, HubSpot, Salesforce): keep as nightly/weekly regression tests to validate real API compatibility
- **Custom request override tests**: these test vendor-specific code, not the framework
- **Real OAuth2 token refresh**: one live test validates the token refresh flow end-to-end

## Expected Outcome

With all 6 stories complete (~40 new unit tests), the Fides unit test suite covers the same framework behavior that ~130 of the 301 fidesplus integration tests currently validate. This enables:

1. **Reclassify per-vendor integration tests as optional/nightly** since framework correctness is proven by unit tests
2. **Catch framework regressions in seconds** (unit tests) instead of minutes (integration tests against real APIs)
3. **Reduce credential maintenance burden** by not requiring 92 active vendor accounts for CI
4. **Improve developer velocity** by giving fast feedback on SaaS connector changes
